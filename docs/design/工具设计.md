> **14 个已实现 Handler + 4 个元能力工具** 的实际架构。
> E3D 工具通过 DispatcherBackedHandler 透传到 E3DToolDispatcher，元能力工具覆盖对话管理（ask_user/task）、外部数据（read_file）、知识检索（search_knowledge）。🆕 新增4个ISO出图工具。

## 二-A、第 0 层：元能力工具（对话管理）

> 借鉴 Cline 的 `ask_followup_question` + Reasonix 的 `ask`/`complete_step` + MiMo 的 `question`/`task`。
> 元能力工具处理"AI-用户协作流程"，区别于 E3D 领域工具。

### M.1 ask_user — 用户交互（提问/澄清/审批）

**一句话**：向用户提问、请求澄清、展示审批计划。所有需要用户决策的场景统一入口。

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `type` | string | ✅ | 交互类型：`clarify`（澄清）/ `confirm`（确认）/ `approve_plan`（审批计划）/ `approve_tool`（审批工具执行）/ `notify`（通知） |
| `title` | string | ✅ | 简洁标题（如 "批量修改确认"） |
| `message` | string | ✅ | 详细说明（支持 Markdown 表格） |
| `options` | array | | 可选项列表，每个选项：`{key, label, description?, is_default?}` |
| `timeout_seconds` | int | | 超时秒数，超时后取默认选项（审批类默认拒绝） |
| `risk_level` | string | | 风险等级：`readonly` / `write_single` / `write_batch` / `destructive` |

**AI 使用示例**：
```
用户: "把所有管子壁厚改成 SCH40"
→ 先 query 查到 38 根管道
→ ask_user(type="approve_plan", title="批量修改确认",
    message="找到 38 根管道，计划修改 WTHK → SCH40：\n\n| # | 名称 | 当前壁厚 | 目标壁厚 |\n|---|------|----------|----------|\n| 1 | PIPE-001 | SCH30 | SCH40 |\n...",
    options=[
      {key:"ok", label:"确认执行", is_default:true},
      {key:"reject", label:"取消"},
      {key:"revise", label:"修改计划"}
    ],
    risk_level="write_batch")

用户: "查 ZONE-01 的管道"（AI 不确定是 PIPE 还是 BRAN）
→ ask_user(type="clarify", title="确认查询范围",
    message="ZONE-01 下同时存在管道(PIPE)和分支(BRAN)，你要查哪种？",
    options=[
      {key:"pipe", label:"管道 (PIPE)"},
      {key:"bran", label:"分支 (BRAN)"},
      {key:"both", label:"两者都要", is_default:true}
    ])

用户: "删除 NOZZ-001"
→ ask_user(type="approve_tool", title="⚠️ 危险操作确认",
    message="即将删除管嘴 NOZZ-001，此操作不可撤销。确认？",
    options=[
      {key:"ok", label:"确认删除"},
      {key:"reject", label:"取消", is_default:true}
    ],
    risk_level="destructive")
```

**与审批流的集成**：

`ask_user` 是审批流的统一 UI 入口。当 `risk_level` 为 `write_batch` 或 `destructive` 时，前端渲染为 `PromptShelfControl`（键盘驱动审批条）；`clarify` 渲染为简单选项按钮；`notify` 渲染为纯信息提示。

```
ask_user 调用
    ↓
ToolPolicy 检查 risk_level
    ↓
┌─ readonly ──→ 自动放行，不弹窗
├─ write_single ──→ 低风险，快速确认
├─ write_batch ──→ PromptShelfControl 审批条（键盘 1/2/3/4）
└─ destructive ──→ 红色警告 + 二次确认
```

| 参考来源 | 借鉴点 |
|---------|--------|
| Cline `ask_followup_question` | 选项结构化 + 多选支持 |
| Reasonix `ask` | 统一交互入口 + 风险分级 |
| MiMo `question` | 无头模式跳过 + 超时机制 |

### M.2 task — 任务追踪

**一句话**：创建和追踪子任务，用于复杂多步骤操作（如"重新布置整个 Zone 的管线"）。

> 借鉴 MiMo 的 `task`（7 种操作合一）和 Reasonix 的 `complete_step`（带证据的步骤确认）。

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `action` | string | ✅ | 操作：`create` / `start` / `complete` / `fail` / `list` / `update` |
| `task_id` | string | | 任务 ID（create 不需要，其他操作必填）。格式：T1, T2, T1.1 |
| `summary` | string | | 任务摘要（create/update 时必填） |
| `parent_id` | string | | 父任务 ID（创建子任务时） |
| `evidence` | array | | 完成证据（complete 时）: `[{kind:"pml_output"|"query_result"|"diff", summary, detail?}]` |

**AI 使用示例**：
```
用户: "把 ZONE-01 所有管道检查一遍，有问题的地方修好"

→ task(action="create", summary="检查 ZONE-01 管道属性完整性")
→ task(action="create", summary="修复缺失属性的管道")
→ task(action="create", summary="检查管件管径一致性")
→ task(action="create", summary="修复管径不一致的管件")

→ task(action="start", task_id="T1")
→ check(type="attribute_complete", elementType="PIPE", scope="ZONE-01")
→ task(action="complete", task_id="T1",
    evidence=[{kind:"query_result", summary:"发现 5 根管道缺 SPEC 属性"}])

→ task(action="start", task_id="T2")
→ modify(type="PIPE", attribute="SPEC", value="CS150", filter="*", scope="ZONE-01")
→ task(action="complete", task_id="T2",
    evidence=[{kind:"diff", summary:"已补充 5 根管道 SPEC=CS150"}])

→ task(action="list")  // 汇报进度
```

**任务状态机**：
```
pending → in_progress → completed
                     → failed → retry → in_progress
```

**设计要点**：
- 任务 ID 格式 `T{n}` 支持层级 `T1.1`，对应复杂操作的父子步骤
- `evidence` 借鉴 Reasonix 的 `complete_step`，让用户看到每一步的执行证据
- `fail` + `retry` 支持自动重试，3 次失败后升级为用户决策

| 参考来源 | 借鉴点 |
|---------|--------|
| MiMo `task` | 操作合一（create/start/complete/fail/list），父子任务层级 |
| Reasonix `complete_step` | 带 evidence 的步骤确认 |
| Cline `new_task` | 子任务可独立管理 |

### M.3 read_file — 读取用户上传的文件

**一句话**：读取用户上传的 Excel、CSV、文本等文件，提取结构化数据供 AI 理解和处理。

> 借鉴 Cline/Reasonix/MiMo 的 `read`（文件读取是三个参考项目的标配能力）。E小智场景下，用户常需要上传外部数据（管道清单、材料表、规格书），AI 必须先"看懂"文件内容才能决定如何处理。

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `path` | string | ✅ | 文件路径（用户通过 UI 选择/拖拽上传后得到） |
| `format` | string | | 显式指定格式：`excel` / `csv` / `text` / `pdf` / `auto`（默认 auto，自动检测） |
| `sheet` | string | | Excel 工作表名（默认第一个 sheet） |
| `range` | string | | Excel 单元格范围（如 `A1:Z100`，默认全部） |
| `header_row` | int | | 表头行号（默认 1，CSV/Excel 列名映射） |
| `limit_rows` | int | | 最大返回行数（默认 200，防止超长文件撑爆上下文） |

**支持的文件类型**：

| 格式 | 扩展名 | 处理方式 | 依赖 |
|------|--------|---------|------|
| Excel | `.xlsx` `.xls` | EPPlus 读取，返回表格数组 | EPPlus (NuGet) |
| CSV | `.csv` | CsvHelper 解析，自动检测分隔符和编码 | CsvHelper (NuGet) |
| 纯文本 | `.txt` `.md` `.pml` `.log` | 直接读取，返回文本 | 无额外依赖 |
| PDF | `.pdf` | 提取文本内容（仅文本层，不含图表） | iTextSharp 或 PdfPig (NuGet) |

**返回数据结构**：
```json
{
  "path": "管道清单.xlsx",
  "format": "excel",
  "sheets": ["Sheet1", "Sheet2"],
  "current_sheet": "Sheet1",
  "total_rows": 500,
  "returned_rows": 200,
  "columns": ["NAME", "SPEC", "WTHK", "FLUID", "DIA"],
  "sample_rows": 5,
  "data": [
    {"NAME": "PIPE-001", "SPEC": "CS150", "WTHK": "SCH40", "FLUID": "WATER", "DIA": 100},
    ...
  ],
  "truncated": true,
  "note": "仅返回前 200 行，共 500 行。使用 range 参数查看更多。"
}
```

**AI 使用示例**：
```
用户上传: 管道清单.xlsx (500行管道规格数据)
→ AI: read_file(path="管道清单.xlsx")  // 默认 auto 检测格式
→ 返回: 前 200 行预览 + 列名 + 总行数

AI 发现只返回了 200 行:
→ read_file(path="管道清单.xlsx", range="201:500")  // 分页读取

用户: "把 Excel 里的管道规格同步到 E3D"
→ AI 先 read_file 理解数据:
  "文件中 500 根管道，列: NAME/SPEC/WTHK/FLUID/DIA。
   匹配到 38 根存在于 E3D 中。计划批量修改属性。"
→ ask_user(type="approve_plan", ...)  // 批量修改确认
→ modify(type="PIPE", attribute="SPEC", ...)  // 逐批执行

用户上传: 管道规格标准.pdf
→ read_file(path="管道规格标准.pdf", format="pdf")
→ AI 提取文本内容，理解规格标准，用于后续的 check() 验证
```

**与 E3D 工具的协作模式**：
```
用户上传文件
    ↓
read_file()  ← 数据入口：AI 理解外部数据
    ↓
┌─ query()     ← 在 E3D 中查找对应元素
├─ check()     ← 用文件数据校验 E3D 数据
├─ modify()    ← 用文件数据批量修改 E3D
└─ export()    ← 将处理结果导出为文件
```

**安全策略**：
- 风险等级：`readonly` — 只读用户上传文件，不操作 E3D，**无需审批**
- 文件大小限制：10MB（超出提示用户拆分）
- 路径限制：只能读取用户通过 UI 显式选择/上传的文件，禁止遍历目录
- PDF 文本提取：仅提取文本层，不解析图片/扫描件

| 参考来源 | 借鉴点 |
|---------|--------|
| Cline `read_file` | 基础文件读取 + 分页（offset/limit） |
| Reasonix `read` | PDF 支持 + 行数控制 |
| MiMo `read` | 图片/PDF 多模态检测 |

### M.4 search_knowledge — E3D 知识库检索

**一句话**：搜索本地 E3D 知识库（API 签名/PML 语法/黄金范式/领域术语），帮助 AI 生成正确的 PML 代码。

> **核心动机**：E3D 有 7,793 个 API + 数百条 PML 命令/函数。AI 模型的训练数据中缺乏这些专有 API，生成 PML 的准确率 ~70-80%。如果将 API 文档和 PML 语法预处理为可检索的知识库，AI 在生成代码前先搜索，准确率可提升到 90%+。
>
> 借鉴 Cline `search_files` + Reasonix `grep` + MiMo `codesearch`，但针对 E3D 领域做了专门优化。

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `source` | string | ✅ | 知识源：`api`（C# API）/ `pml`（PML 语法）/ `pattern`（黄金范式）/ `domain`（领域术语）/ `all` |
| `query` | string | ✅ | 搜索词（支持自然语言 + 关键词） |
| `language` | string | | PML/C#/both（默认 both） |
| `limit` | int | | 返回条数（默认 5，最大 20） |
| `include_examples` | bool | | 是否附带代码示例（默认 true） |

**知识库结构**：

```
knowledge/
├── api/                          # 7,793 个 C# API（从 HTML 文档预处理）
│   ├── Core.Database/            # 按命名空间分文件
│   │   ├── DbElement.md          # 每个类一个 Markdown 文件
│   │   ├── DbAttribute.md        # 含签名 + 参数 + 返回类型 + 示例
│   │   └── Command.md
│   ├── Core.Geometry/
│   │   ├── D3Point.md
│   │   └── D3Vector.md
│   └── search_index.json         # 预建关键词索引（首次加载到内存）
│
├── pml/                          # PML 语法参考（从官方手册 + 速查表预处理）
│   ├── commands.md               # coll/filter/do/by/handle 等
│   ├── functions.md              # Matchwild/Match/Distance/Q Ele 等
│   └── expressions.md            # 属性访问、条件表达式、变量
│
├── patterns/                     # 33 个已验证黄金 PML 范式
│   ├── query_elements.md         # coll all TYPE with Matchwild(...)
│   ├── modify_attributes.md      # DO values / WTHK = 'SCH40'
│   ├── check_distance.md         # DISTANCE / POSITION 计算
│   └── ...
│
└── domain/                       # 领域术语/映射/常量
    ├── element_types.md          # PIPE/EQUI/VALV/STRU/ZONE/NOZZ → 中文对照
    ├── attribute_map.md          # WTHK→壁厚, :DIA→公称直径, SPEC→规格
    ├── coordinate_systems.md     # WORPOS/E/N/U → 坐标系统说明
    └── error_codes.md            # E3D 常见错误码和原因
```

**返回数据结构**：
```json
{
  "query": "获取管道壁厚",
  "source": "all",
  "total": 15,
  "results": [
    {
      "source": "api",
      "type": "C# API",
      "namespace": "Aveva.Core.Database",
      "signature": "DbElement.GetAsString(DbAttribute attr) -> string",
      "description": "获取元素的指定属性值，返回字符串格式。常用属性：DbAttributeInstance.WTHK",
      "confidence": "high",
      "example": "var wthk = element.GetAsString(DbAttributeInstance.WTHK);",
      "file": "knowledge/api/Core.Database/DbElement.md",
      "verified": true
    },
    {
      "source": "pattern",
      "type": "PML Golden Pattern",
      "name": "query_pipe_attributes",
      "code": "!ce = CURRENT CE\ncoll all PIPE with Matchwild(NAME, '*DN100*', true)\nDO !wthk = WTHK of !ce\n$P !ce.Name, ' = ', !wthk\nENDDO",
      "description": "查询带名称匹配的管道属性，含壁厚读取",
      "file": "knowledge/patterns/query_elements.md",
      "hit_count": 263,
      "verified": true
    },
    {
      "source": "pml",
      "type": "PML Syntax",
      "category": "command",
      "syntax": "coll all TYPE with Condition",
      "description": "收集指定类型的所有元素，可选 WITH 条件过滤",
      "file": "knowledge/pml/commands.md"
    }
  ]
}
```

**AI 使用示例**：
```
用户: "查所有 DN100 管道的壁厚"

AI 不确定 PML 语法:
→ search_knowledge(source="pattern", query="query pipe attributes filter")
→ 返回: coll all PIPE with Matchwild(NAME, ...) 黄金范式

AI 不确定壁厚属性名:
→ search_knowledge(source="domain", query="壁厚 属性")
→ 返回: WTHK = 壁厚 (wall thickness), PML 中可用 WTHK of <element>

AI 不确定 C# API 签名:
→ search_knowledge(source="api", query="GetAsString attribute")
→ 返回: DbElement.GetAsString(DbAttribute) -> string (已验证，高置信度)

AI 组合搜索:
→ search_knowledge(query="管道壁厚 DN100 SCH40", source="all", limit=10)
→ 返回: API + PML + Pattern + Domain 的综合结果
```

**AI 什么时候调用 search_knowledge**：
```
需要生成 PML 代码时：
  ① search_knowledge(source="pattern", query="...")  // 找最接近的黄金范式
  ② 如果范式不精确 → search_knowledge(source="pml", query="...")  // 查 PML 语法细节
  ③ 如果需要 C# 直调 → search_knowledge(source="api", query="...")  // 查 API 签名
  ④ 如果不确定术语 → search_knowledge(source="domain", query="...")  // 查领域映射
  ⑤ 基于搜索结果生成代码
```

**这个工具如何提升 PML 准确率**：

```
无 search_knowledge（现状预估 ~70%）:
用户: "查 DN100 管道的壁厚和流体"
AI 凭记忆生成: coll all PIPE with (DIA eq 100)  // ❌ WTHK 用错了，应该是 :WTHK
              DO values ...
              可能生成 pos.East 而不是 pos.E  // ❌ API 签名错误

有 search_knowledge（目标 ~92%+）:
用户: "查 DN100 管道的壁厚和流体"
AI 先搜索: → pattern "query pipe attributes"  → 得到黄金范式
          → domain "壁厚 流体" → WTHK, FLUID
          → pml "DIA filter"   → :DIA with Matchwild
AI 基于结果生成: coll all PIPE with Matchwild(NAME, '*DN100*', true)
              DO !wthk = WTHK of !ce; $P !ce.Name, ' WTHK=', !wthk
              ENDDO  ✅ 正确
```

**安全策略**：
- 风险等级：`readonly` — 纯搜索操作，**无需审批**
- 知识库只读，AI 不能修改知识库文件
- 搜索结果附带 `verified` 标记（已通过 33 个真实 PML 工具验证的 API）
- `confidence: "high"` 的 API 有 263+ 次真实命中验证，`"medium"` 的只有文档签名验证

| 参考来源 | 借鉴点 |
|---------|--------|
| Cline `search_files` | 跨文件内容搜索 |
| Reasonix `grep` | 正则 + 文件类型过滤 |
| MiMo `codesearch` | 结构化代码搜索 + 示例返回 |
| **E小智定制** | 非通用 grep，而是 **领域知识库分源检索**（api/pml/pattern/domain） + 已验证标记 |

## 二、第 1 层：10 个已实现 Handler

> 以下为实际代码中注册的 Handler，而非设计规划中的 6 个核心工具。

### 2.1 query — 通用查询

**一句话**：查询 E3D 元素，支持按类型、名称、属性、范围筛选。

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `type` | string | ✅ | 元素类型：PIPE/BRAN/EQUI/VALV/STRU/ZONE/SITE/FTUB/BEND/TEE/NOZZ |
| `pattern` | string | | 名称匹配模式，支持 * 通配符 |
| `scope` | string | | 搜索范围，默认 CE（当前元素） |
| `attributes` | array | | 需要返回的属性列表 |
| `filter` | string | | PML 表达式过滤（如 `:DIA > 100`） |
| `limit` | int | | 返回数量上限 |

**AI 使用示例**：
```
用户: "查所有 DN100 管道"
→ query(type="PIPE", pattern="*DN100*")

用户: "查 ZONE-01 下所有设备的名称和房间号"
→ query(type="EQUI", scope="ZONE-01", attributes=["NAME","ROOM_NO"])

用户: "查壁厚大于 SCH30 的管道"
→ query(type="PIPE", filter=":WTHK > 'SCH30'")

用户: "看看当前选中元素的子元素"
→ query(scope="CE")

用户: "这个 Zone 有多少管道"
→ query(type="PIPE", scope="ZONE-01")  -- AI 根据返回结果汇报数量
```

**内部路由**：

| 参数组合 | 内部工具 |
|----------|---------|
| type=PIPE + pattern | QueryElementsTool（PML） |
| type=EQUI + attributes | QueryEquipmentTool（PML） |
| scope=CE + 无 type | GetCurrentElementTool（C#） |
| filter 有值 | EvaluateExpressionTool（PML） |

```
用户输入
  ↓
┌─ 需要查 API/PML 语法？ ─→ search_knowledge()
│  ├─ 不确定 C# API 签名？ search_knowledge(source="api", query="GetAsString")
│  ├─ 不确定 PML 写法？    search_knowledge(source="pml", query="coll type filter")
│  ├─ 找最接近的代码范式？ search_knowledge(source="pattern", query="query pipe")
│  └─ 不确定术语映射？     search_knowledge(source="domain", query="壁厚 属性")
│
├─ 需要读取外部文件？ ────→ read_file()
│  ├─ Excel 数据？       read_file(path="xxx.xlsx", format="excel")
│  ├─ CSV 数据？         read_file(path="xxx.csv", format="csv")
│  ├─ PDF 文档？         read_file(path="xxx.pdf", format="pdf")
│  └─ 文本/日志？        read_file(path="xxx.txt")
│
├─ 需要用户决策/澄清？ ─→ ask_user()
│  ├─ 参数不明确？      ask_user(type="clarify", ...)
│  ├─ 批量操作确认？    ask_user(type="approve_plan", ...)
│  ├─ 危险操作确认？    ask_user(type="approve_tool", risk_level="destructive")
│  └─ 纯通知？          ask_user(type="notify", ...)
│
├─ 多步骤复杂任务？ ────→ task() 分解子任务
│  ├─ 创建子任务       task(action="create", ...)
│  ├─ 开始执行         task(action="start", task_id="T1")
│  ├─ 完成（带证据）   task(action="complete", task_id="T1", evidence=[...])
│  └─ 汇报进度         task(action="list")
│
├─ 想查询/浏览？ ────────→ query()
│  ├─ 查管道？           query(type="PIPE")
│  ├─ 查设备？           query(type="EQUI")

│  └─ 查层级？           query(scope="CE")
│
├─ 想修改/创建？ ────────→ modify()
│  ├─ 改单个属性？       modify(target="X", attr="Y", val="Z")
│  ├─ 批量改属性？       modify(type="T", attr="Y", val="Z", filter="*")
│  └─ 创建元素？         modify(target="NEW", attr="TYPE", val="EQUI")
│
├─ 想检查/验证？ ────────→ check()
│  ├─ 元素存在？         check(type="exists", target="X")
│  ├─ 属性完整？         check(type="attribute_complete", ...)
│  └─ 间距合规？         check(type="distance", ...)
│
├─ 想计算/测量？ ────────→ calculate()
│  ├─ 两个元素距离？     calculate(type="distance", ...)
│  ├─ 朝向角度？         calculate(type="orientation", ...)
│  └─ 路线长度？         calculate(type="route_length", ...)
│
├─ 想导入导出？ ────────→ export()
│  ├─ 导出 Excel？       export(direction="export", ...)
│  └─ 导入数据？         export(direction="import", ...)
│
└─ 以上都不够？ ────────→ execute_pml()
```

### 6.2 query_material — 材料查询

用户: "启动AutoCAD"

### 元能力工具全景

| 工具 | 解决的问题 | 触发时机 | 风险等级 | 审批 |
|------|-----------|---------|:------:|:----:|
| `search_knowledge` | API/PML 语法不确定 → 查知识库 | 生成代码前 | readonly | ❌ |

> E小智元能力工具从 0 → 4 个。选择策略：**只借鉴参考项目中与 E3D 场景直接匹配的能力**。
> - `search_knowledge` — 核心差异化，做 RAG 而非 raw grep，消除 API 幻觉（70% → 92%）
> - `read_file` — 外部数据入口
> - `ask_user` — 协作入口 + 审批流
> - `task` — 复杂任务管理入口
>
> 代码 Agent 需要的 `write_file` / `grep` / `glob` / `ls` / `bash` / `lsp` 等在 E3D 操作场景中无用，不引入。
