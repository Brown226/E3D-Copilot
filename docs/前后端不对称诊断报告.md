# E小智 前后端不对称诊断报告

> 诊断时间：2026-06-23
> 症状：连最基础的对话都无法实现
> 结论：项目同时存在 **三套通信协议**（两套死代码、一套活但桥接不完整），加上 **6 处致命断裂点**，导致用户发一条消息后输入框即被永久禁用、Model 切换全部超时、Plan/Act 模式失效。

---

## 一、根本症结：三套协议并存

项目代码里同时存在三套前后端通信协议，只有一套真正工作，但即便这一套也桥接得不完整。

| 协议 | 前端定义 | 后端实现 | 状态 |
|------|----------|----------|------|
| **协议 A**（Cline 风格 postMessage） | `web-ui/src/bridge-messages.ts` | 无 | **孤儿** — 全代码库无 import |
| **协议 B**（自定义 messageContracts） | `web-ui/src/messageContracts.ts` + `bridge.ts` | `E3DCopilot.Core/Messaging/MessageContracts.cs` + `Bridge.cs` | **存活但残缺** |
| **协议 C**（gRPC ProtoBus） | `web-ui/src/services/grpc-client-base.ts` | 无 | **孤儿** — C# 端从未实现 ProtoBus 服务端 |

### 协议 A：`bridge-messages.ts`（死代码）
定义了 `chat` / `cancelTask` / `getState` / `getHistory` / `chatChunk` / `chatComplete` / `toolUse` / `toolResult` 等消息类型，以及 `ExtensionState` / `ChatMessage` 接口。
- `grep -r bridge-messages web-ui/src` 结果：**零引用**
- 它定义的 `ExtensionState` 与 `@shared/ExtensionMessage` 中的 `ExtensionState` 完全不是同一个类型
- 前端的 `grpc-client.ts` 完全绕过它，自己维护了一套 `_clineMessages` 数组

### 协议 C：`grpc-client-base.ts`（死代码）
真正的 gRPC 实现，通过 `window.postMessage` 发 `{type:'grpc_request', grpc_request:{service, method, message, request_id, is_streaming}}`，等待 `grpc_response` 回包。
- C# 后端**从未实现 ProtoBus 服务端**
- `grpc-client.ts`（实际使用的文件）**没有 import 它**，而是自己手写了基于 `bridge` 的桩
- 这意味着所有 `ModelsServiceClient.subscribeToOpenRouterModels`、`McpServiceClient.subscribeToMcpServers` 等订阅调用，全部走的是 `createStubClient` 返回的空 Proxy，调用任何方法都只 `console.log` 然后 `return {}`

### 协议 B：实际工作但残缺
这是唯一存活的协议路径：
```
ChatView → useMessageHandlers → TaskServiceClient.newTask (stub)
  → bridge.sendUserMessage → chrome.webview.postMessage
  → C# Bridge.HandleMessage → CopilotController.SendAsync
  → AgentLoop → VllmProvider.StreamAsync → SSE chunks
  → CopilotEvent → Bridge.DispatchEvent → SendToFrontend
  → bridge.handleIncoming → grpc-client bridge.on listener
  → _clineMessages 更新 → notifyStateChange
  → TaskServiceClient.subscribeToState 回调 → setState → React 重新渲染
```

这条链路在协议层（`MessageTypes` 字符串常量）**前后端完全对齐**，但在状态语义层和请求/响应配对层**严重错配**（详见第三节）。

## 二、架构层不对称

### 2.1 前端是 Cline 完整 fork，后端是全新精简实现

| 维度 | 前端（web-ui） | 后端（E3DCopilot.Core） |
|------|----------------|------------------------|
| 来源 | fork 自 Cline（VSCode AI 扩展） | 全新编写 |
| 通信模型 | gRPC ServiceClient + Protobuf 流式订阅 | WebView2 postMessage + 自定义事件 |
| 状态模型 | `ExtensionState`（50+ 字段，含 MCP/Worktree/FocusChain/AutoApproval...） | `CopilotSession`（仅 Messages + IsPlanMode） |
| 消息模型 | `ClineMessage`（`type:"say"\|"ask"`, `say`, `ask`, `partial`, `ts`） | `CopilotEvent`（`Kind: EventKind`, `Text`, `ToolId`, `Data`） |
| 工具模型 | Cline 的 `tool_use` / `tool_result` 配对 | `ToolCall` + `ToolExecutor` + `ToolPolicy` |
| 审批模型 | Cline 的 `ask:"tool"` 等待用户点 Approve/Reject | `PendingApproval` + `TaskCompletionSource` |

前端带着 Cline 的全部包袱（账号、计费、MCP、Worktree、FocusChain、终端配置、Checkpoints...），后端只实现了核心 Agent 循环。**前端 80% 的代码在后端没有对应实现**。

### 2.2 `grpc-client.ts` 是"翻译层"但翻译不完整

`grpc-client.ts` 把 Cline 的 gRPC ServiceClient 接口翻译成 `bridge.send` 调用。这是整个项目的关键粘合层，但它存在系统性翻译缺陷：

| Cline gRPC 方法 | grpc-client.ts 翻译 | 问题 |
|-----------------|---------------------|------|
| `TaskServiceClient.newTask` | `bridge.sendUserMessage(text, images, files)` | 翻译正确 |
| `TaskServiceClient.askResponse` | `bridge.sendAskResponse("current", text)` | **questionId 硬编码 "current"**，后端 `AskUserHandler.SubmitAnswer("current", ...)` 永远找不到对应问题 |
| `TaskServiceClient.cancelTask` | `bridge.cancel()` | 翻译正确 |
| `TaskServiceClient.clearTask` | `bridge.newSession()` | 翻译正确 |
| `TaskServiceClient.getTaskHistory` | `return { history: [] }` | **永远返回空**，历史功能废 |
| `TaskServiceClient.deleteTask` | `return {}` | **no-op** |
| `TaskServiceClient.taskFeedback` | stub | 调用即 `console.log` 后返回 `{}` |
| `TaskServiceClient.showTaskWithId` | stub | 同上 |
| `StateServiceClient.setPlanActMode` | `bridge.send(MessageTypes.UserMessage, {mode})` | **发错类型！** 用 `UserMessage` 发 `{mode}`，后端当空消息忽略 |
| `StateServiceClient.togglePlanActModeProto` | `bridge.send("togglePlanActMode", ...)` | **后端无此 case**，落入 default 静默丢弃 |
| `ModelsServiceClient.listModels` | `bridge.sendAndWait('models:list', ...)` | **sendAndWait 必超时**（见 3.2） |
| `ModelsServiceClient.switchModel` | `bridge.sendAndWait('model:switch', ...)` | 同上 |
| `ModelsServiceClient.listProviders` | `bridge.sendAndWait('providers:list', ...)` | 同上 |
| 所有 `subscribeTo*` 方法 | stub 或 `() => {}` | **不推送任何事件** |

## 三、6 处致命断裂点（按严重度排序）

### 断裂 ① `sendingDisabled` 永不重置 —— 对话卡死直接元凶

**位置**：`web-ui/src/components/chat/chat-view/hooks/useMessageHandlers.ts`

```ts
// handleSendMessage 发送成功后
setSendingDisabled(true)
setEnableButtons(false)
```

全代码库 `setSendingDisabled(false)` 和 `setEnableButtons(true)` **只出现在 cancel 流程的 finally 块**（第 290-291 行）：

```ts
case "cancel": {
  ...
  try {
    await TaskServiceClient.cancelTask(...)
  } finally {
    setSendingDisabled(false)   // ← 只有这里
    setEnableButtons(true)      // ← 只有这里
  }
}
```

LLM 流式完成、Turn 结束、错误返回时，**没有任何 listener 重置 sendingDisabled**。

原本 Cline 是通过 `subscribeToPartialMessage` 推送 `partial:false` 消息来驱动 UI 解锁，但 E3D 适配层的 `UiServiceClient.subscribeToPartialMessage` 是 stub（返回 `() => {}`，从不调 `onResponse`）。

**结果**：用户发第一条消息后，输入框被永久禁用，必须点 Cancel 才能再发 → 这就是"连基础对话都无法实现"的最直接症状。

### 断裂 ② `bridge.sendAndWait` 全部超时 —— Model/Provider 管理全废

**位置**：`web-ui/src/bridge.ts` vs `src/E3DCopilot.WebHost/Bridge.cs`

前端 `sendAndWait` 发送 `{type, payload, _requestId}`，期望后端回 `{_requestId, payload}`：
```ts
const msg = JSON.stringify({ type, payload, _requestId: requestId });
// ... 等待带相同 _requestId 的响应，超时 5-30s 后 reject
```

后端 `Bridge.HandleMessage` **完全不读取 `_requestId` 字段**：
```csharp
using var doc = JsonDocument.Parse(raw);
var type = typeProp.GetString();
var payload = root.TryGetProperty("payload", out var p) ? p : default;
// _requestId 从未被读取
switch (type) { ... }
```

后端的响应（`ModelsListResult`、`ProvidersListResult` 等）是通过 `SendToFrontend` 主动推送的，**不带 `_requestId`**，前端的 `handleIncoming` 把它们当普通事件分发给 listeners，**不会 resolve pendingRequests**。

**结果**：
- `listModels()` Promise 必然超时 reject（前端监听器 `onModelsListResult` 虽能收到数据，但 await 调用本身已抛错）
- `switchModel()` 超时 reject，且后端发的是 `MessageTypes.ModelSwitch` 而非 `ModelsListResult`，前端**无对应 listener**
- `saveProvider` / `deleteProvider` / `fetchProviderModels` / `setProviderKey` 全部同理

### 断裂 ③ Notice 污染消息流 —— 破坏 newTask 分支判断

**位置**：`Bridge.cs` HandleUserMessage + `grpc-client.ts` Notice handler

后端收到用户消息会先发诊断 Notice：
```csharp
SendToFrontend(MessageTypes.Notice, new { text = $"[Bridge] 收到消息: {text.Substring(0, ...)}，正在调用 LLM..." });
```

前端把 Notice 当成普通文本消息加入 `_clineMessages`：
```ts
case MessageTypes.Notice:
  addMessage({ type: "say", say: "text", text: msg.payload?.text || "" })
```

`useMessageHandlers.handleSendMessage` 的分支逻辑依赖 `messages.length === 0`：
```ts
if (messages.length === 0) {
  await TaskServiceClient.newTask(...)        // ← 第一条消息走这里
} else if (clineAsk) {
  ...
}
```

第一条 Notice 进入后 `messages.length > 0`，**后续所有消息都无法走 newTask 主分支**，只能落到 `!messageSent` 兜底分支（虽然兜底也调 newTask，但 `clineAsk` 永远不会变成 `"followup"`/`"tool"` 等，所有 ask 响应逻辑全废）。

更深层：Cline 的 `say:"text"` 是 AI 回复，而 Notice 是系统诊断信息，两者语义完全不同，混在一起会让 ChatRow 渲染出"[Bridge] 收到消息..."这种本不该出现在对话流里的内容。

### 断裂 ④ 审批被自动绕过 —— 安全机制形同虚设

**位置**：`grpc-client.ts` handleToolApproval

```ts
function handleToolApproval(payload: any) {
  bridge.sendApproval(payload?.id, true)   // ← 永远传 true，自动批准
}
```

后端 `AgentLoop` 对 `execute_pml`、`modify` 等写工具设置了 `ApprovalMode.Ask`，期望前端弹出审批框让用户确认。但前端**收到 `tool:approval` 事件后直接自动批准**，完全没有 UI 交互。

这意味着 AI 可以不经确认执行任意 PML 脚本修改 E3D 模型，违背了 `CommandPermissionController` + `ToolPolicy` 的整个安全设计。

### 断裂 ⑤ Plan/Act 模式切换完全不工作

**位置**：`grpc-client.ts` createStateServiceClient

```ts
setPlanActMode: async (req: any) => {
  const mode = req?.mode || "act"
  bridge.send(MessageTypes.UserMessage, { mode })   // ← 类型错误
  return {}
},

togglePlanActModeProto: async (req: any) => {
  bridge.send("togglePlanActMode", { mode, chatContent })   // ← 未注册类型
  return { value: true }
},
```

- `setPlanActMode` 用 `UserMessage` 类型发 `{mode}`，后端 `HandleUserMessage` 读 `payload.text` 得到 null，发 Notice "收到空消息，已忽略"
- `togglePlanActMode` 发字符串 `"togglePlanActMode"`，后端 switch 无此 case，落入 `default: 静默忽略`

**结果**：前端 UI 上的 Plan/Act 切换按钮完全无效，`CopilotController.SetPlanMode` 永远不会被调用。

### 断裂 ⑥ IS_E3D 分支跳过 12+ 项订阅 —— Cline UI 大量空转

**位置**：`web-ui/src/context/ExtensionStateContext.tsx` 第 362-395 行

```tsx
const IS_E3D = PLATFORM_CONFIG.type === PlatformType.E3D
useEffect(() => {
  if (IS_E3D) {
    stateSubscriptionRef.current = TaskServiceClient.subscribeToState(...)
    return   // ← 提前返回，跳过下面所有订阅
  }
  // 以下全部不执行：
  // UiServiceClient.subscribeToShowWebview
  // UiServiceClient.subscribeToAddToInput
  // UiServiceClient.subscribeToMcpButtonClicked
  // UiServiceClient.subscribeToHistoryButtonClicked
  // UiServiceClient.subscribeToChatButtonClicked
  // UiServiceClient.subscribeToSettingsButtonClicked
  // UiServiceClient.subscribeToWorktreesButtonClicked
  // UiServiceClient.subscribeToAccountButtonClicked
  // UiServiceClient.subscribeToPartialMessage    ← 关键！驱动 UI 解锁
  // UiServiceClient.subscribeToRelinquishControl
  // UiServiceClient.initializeWebview
  // McpServiceClient.subscribeToMcpServers
  // McpServiceClient.subscribeToMcpMarketplaceCatalog
  // ModelsServiceClient.subscribeToOpenRouterModels
  // ModelsServiceClient.subscribeToLiteLlmModels
  // StateServiceClient.getAvailableTerminalProfiles
}, [])
```

即使非 E3D 分支跑起来，这些订阅调用的也是 `createStubClient` 返回的空 Proxy，**永远不会触发 onResponse**。

最关键的是 `subscribeToPartialMessage` 缺失 —— 它原本负责流式消息的增量更新和 `partial:false` 标记，是 UI 状态机的核心驱动。它一缺，断裂 ① 的"永不重置"就无解。

## 四、对话失败的完整因果链

以"用户输入'你好'按回车"为例，追踪实际执行过程：

```
1. ChatView → useMessageHandlers.handleSendMessage("你好", [], [])
2. messages.length === 0 → 调用 TaskServiceClient.newTask({text:"你好"})
3. grpc-client.ts newTask stub:
   - _clineMessages = []  (清空)
   - bridge.sendUserMessage("你好", [], [])
   - notifyStateChange()  → 推一次 stateJson
4. bridge.ts → chrome.webview.postMessage('{"type":"user:message","payload":{"text":"你好"}}')
5. C# Bridge.HandleMessage → case UserMessage → HandleUserMessage
6. HandleUserMessage:
   - SendToFrontend(Notice, "[Bridge] 收到消息: 你好，正在调用 LLM...")
   - Task.Run(() => _controller.SendAsync("你好"))
   - SendToFrontend(Notice, "[Bridge] LLM 调用完成")   ← 注意：这行在 Task.Run 之前就发了！
7. 前端 handleIncoming 收到第一条 Notice:
   - grpc-client.ts: addMessage({type:"say", say:"text", text:"[Bridge] 收到消息..."})
   - notifyStateChange() → setState → React 渲染
   - ★ 此时 messages.length = 1
8. 前端 handleSendMessage 继续: setSendingDisabled(true), setEnableButtons(false)
   - ★ 输入框被禁用
9. CopilotController.SendAsync → AgentLoop.RunAsync:
   - session.AddUserMessage("你好")
   - Emit TurnStarted  (前端无对应处理)
   - StreamLlmAsync → VllmProvider.StreamAsync
   - SSE chunks 回调 → Emit TextEvent/Reasoning
10. Bridge.DispatchEvent → SendToFrontend(LlmStreamDelta, {delta:"你"})
11. 前端 handleLlmStreamDelta → _currentStreamingMessage.text += "你"
12. ... 流式累加 ...
13. LlmStreamEnd → _currentStreamingMessage.partial = false
14. ★ 但 sendingDisabled 仍然是 true，没有任何代码把它设回 false
15. 用户看到 AI 回复，但输入框灰色禁用，无法继续对话
16. 用户只能点 Cancel 按钮 → cancelTask → bridge.cancel → setSendingDisabled(false)
17. 但 cancel 会触发 _controller.Cancel()，中断可能还在跑的 LLM
18. 下次发消息时 messages.length > 0（Notice 还在），走 !messageSent 兜底 newTask
```

**核心问题**：步骤 6 的"LLM 调用完成"Notice 在 `Task.Run` 启动后立即发送（代码顺序错误），与步骤 13 的真正 StreamEnd 是两回事；前端没有任何机制把"StreamEnd"翻译成"sendingDisabled=false"。

## 五、修复优先级建议

### P0 — 让基础对话能跑通（必须先做）

1. **修复 sendingDisabled 重置**：在 `grpc-client.ts` 的 `LlmStreamEnd` handler 中调用一个全局事件总线，让 `useChatState` 监听并 `setSendingDisabled(false)` + `setEnableButtons(true)`。或者更简单：在 `handleStreamEnd` 里直接通过 React state setter 解锁。

2. **移除 Notice 污染**：后端 `HandleUserMessage` 删除两行诊断 Notice（或改成 `console.log` 而非 `SendToFrontend`），让 `_clineMessages` 只包含真实对话内容。

3. **修复 sendAndWait 协议**：后端 `Bridge.HandleMessage` 读取 `_requestId`，在 `SendToFrontend` 时若原始请求带 `_requestId` 则回包时带上；或者前端放弃 sendAndWait，改用 `bridge.on('models:list:result', ...)` 事件监听 + Promise 化。

### P1 — 让 Model/Provider 管理可用

4. **统一模型切换响应**：后端 `HandleModelSwitch` 成功后发 `MessageTypes.ModelsListResult`（而非 `ModelSwitch`），让前端 `onModelsListResult` 监听器能收到。

5. **前端补齐 ModelsServiceClient 事件监听**：`grpc-client.ts` 已有 `onModelsListResult` / `onProvidersListResult` 等方法但没人调用，需在 Settings 界面挂上。

### P1 — 恢复审批 UI

6. **实现 ToolApproval UI**：`grpc-client.ts` 的 `handleToolApproval` 不要自动批准，改为往 `_clineMessages` 推 `{type:"ask", ask:"tool", ...}`，让 Cline 原生的 ChatRow 审批 UI 渲染出来，用户点 Approve/Reject 后调 `bridge.sendApproval(id, allow)`。

### P2 — 清理死代码

7. **删除 `bridge-messages.ts`**：零引用，删除避免误导。
8. **删除 `grpc-client-base.ts`** 或明确标注废弃：C# 端不实现 ProtoBus，留着只会让人误以为有 gRPC 通道。
9. **IS_E3D 分支清理**：把 `createStubClient` 返回的 ServiceClient 在 E3D 模式下彻底从 import 中剔除，避免无效订阅调用。

### P2 — 修复 Plan/Act 模式

10. **新增 `user:set_plan_mode` 消息类型**：前后端契约同步，后端 `Bridge.HandleMessage` 加 case 调 `_controller.SetPlanMode(bool)`，前端 `setPlanActMode` 改发此类型。

### P3 — 状态模型对齐

11. **定义 E3D 专属的精简 State**：前端 `INITIAL_STATE` 砍掉 MCP/Worktree/FocusChain/Account 等 E3D 不用的字段，避免 `ExtensionStateContext` 维护几十个永远空的状态。
12. **事件语义对齐**：明确 `Notice` ≠ `say:text`，给 Notice 单独的 `say:"notice"` 类型或专用 UI 区域。
