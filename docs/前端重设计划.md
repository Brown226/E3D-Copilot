## 0. 现状诊断与前置假设

### 0.1 当前 `web-ui/` 的问题

| 问题 | 证据 | 影响 |
|---|---|---|
| 体量过大 | `build/assets/index.js` 6.5MB，`index.css` 106KB | 加载慢、构建慢、宿主压力大 |
| 历史包袱重 | 组件目录 11+ 个，含 Cline 特有的 MCP/History/Onboarding/Browser/Plan-Act | 大量死代码和无效依赖 |
| 状态管理臃肿 | `ExtensionStateContext.tsx` 32KB，50+ 状态字段 | 理解和维护成本高 |
| 兼容层复杂 | `grpc-client.ts` 928 行，模拟 Cline 的 gRPC 服务接口 | 大量 stub 和 Proxy，调试困难 |
| 构建状态不明 | `web-ui-build-error.txt` 不存在，但 `build/` 目录已生成 | 需要重新验证是否能从源码构建 |

### 0.2 可复用资产清单（已核实）

| 资产 | 文件路径 | 复用方式 | 风险 |
|---|---|---|---|
| `bridge.ts` | `src/bridge.ts` | ✅ 整体复用 | 已基于 `chrome.webview`，与 WebView2 宿主兼容 |
| `messageContracts.ts` | `src/messageContracts.ts` | ✅ 整体复用 | 需与 C# `MessageTypes` 保持同步 |

| `ModelSwitcher` | `src/components/chat/ModelSwitcher.tsx` | ⚠️ 改写复用 | 依赖 `grpc-client` 的 provider 状态机 |
| `ProviderSettingsSection` | `src/components/settings/sections/ProviderSettingsSection.tsx` | ⚠️ 大幅改写 | 依赖 `grpc-client`、Section 组件、VSCode CSS 变量 |
| `ChatRow` 工具分组逻辑 | `src/components/chat/ChatRow.tsx`（56KB） | ❌ 不复用 | 过于复杂，按新消息类型重写 |

### 0.3 关键依赖关系

```
bridge.ts
  ↑ 被 grpc-client.ts 使用
  ↑ 将被新的 bridgeService.ts / useBridge.ts 使用

messageContracts.ts
  ↑ 被 bridge.ts 和 grpc-client.ts 使用
  ↑ 将被新的所有服务层使用

grpc-client.ts（当前）
  ↑ 被 ModelSwitcher、ProviderSettingsSection、MarkdownBlock 使用
  ↓ 将在新架构中被替换为 useChatStore + bridgeService
```

### 0.4 设计不变式

1. **单一职责**：`e3d-ui/` 只做 Chat + Provider 设置两件事
2. **无 Cline 上下文**：不引入 `ExtensionStateContext`、i18n、`@shared/proto` 等依赖
3. **无历史包袱**：不实现 History、MCP、Browser、Onboarding、Plan/Act 切换
4. **桥接唯一**：所有状态变更来自 `bridge.on()`，所有用户操作通过 `bridge.send()`
5. **类型安全**：所有消息类型来自 `messageContracts.ts`，新增类型同步到 C#
6. **宿主无关**：UI 不假设是 WebView2、VSCode 还是独立浏览器，仅依赖标准 `window` API

### 0.5 ProviderInfo 前后端字段对照

C# `CopilotConfig.ProviderConfig` 与 TS `ProviderInfo` 的差异，需在 Phase 0 确认处理：

| 字段 | C# `ProviderConfig` | TS `ProviderInfo` | 处理 |
|---|---|---|---|
| `Name` | ✅ | ✅ `name` | 一致 |
| `Kind` | ✅ | ✅ `kind` | 一致 |
| `BaseUrl` | ✅ | ✅ `baseUrl` | 一致 |
| `ApiKey` | ✅ | ✅ `apiKey` | 一致（前端只显示 `keySet` 掩码） |
| `Models` | ✅ `List<string>` | ✅ `models: string[]` | 一致 |
| `DefaultModel` | ✅ | ✅ `default` | 一致 |
| `TimeoutMs` | ✅ 120000 | ❌ 无 | 前端不展示，C# 端保留默认值 |
| `Temperature` | ✅ 0.1 | ❌ 无 | 前端不展示，C# 端保留默认值 |
| `MaxTokens` | ✅ 8192 | ❌ 无 | 前端不展示，C# 端保留默认值 |
| `Enabled` | ❌ 无 | ✅ `enabled` | TS 独有，C# 端视为始终 true |
| `BuiltIn` | ❌ 无 | ✅ `builtIn` | TS 独有，`ProvidersService.IsBuiltIn()` 判断 |

**结论**：`TimeoutMs`/`Temperature`/`MaxTokens` 为 C# 端运行时参数，前端无需暴露。`Enabled`/`BuiltIn` 为前端 UI 控制字段，C# 端 `config:sync` 时补全。

## 2. 消息协议与状态管理

### 2.1 前端消息类型

```tsx
type Message =
  // 用户消息
  | { id: string; role: "user"; text: string; ts: number }
  // AI 流式文本回复
  | { id: string; role: "assistant"; text: string; partial: boolean; ts: number }
  // AI 思考/推理过程
  | { id: string; role: "thinking"; text: string; partial: boolean; ts: number }
  // AI 调用工具
  | { id: string; role: "tool_call"; toolId: string; toolName: string; args: Record<string, unknown>; ts: number }
  // 工具执行结果
  | { id: string; role: "tool_result"; toolId: string; toolName: string; result?: string; error?: string; ts: number }
  // 系统错误
  | { id: string; role: "error"; message: string; ts: number }
```

**补充说明**：
- 每条消息必须有 `id`（UUID），用于流式更新时定位目标消息、工具调用与结果关联
- `args` 类型从 `any` 改为 `Record<string, unknown>`，遵循不变式第 5 条类型安全
- 新增 `role: "thinking"` 支持推理模型（Qwen3.7-Plus 等）的思考过程展示
- `partial: boolean` 标识流式进行中，用于显示闪烁光标

### 2.2 状态管理（useChatStore）

使用 Zustand 实现，避免 Context 性能问题：

```tsx
interface ChatStore {
  // 消息
  messages: Message[];
  
  // 流式状态
  isStreaming: boolean;
  currentAssistantMsgId: string | null;
  currentThinkingMsgId: string | null;
  
  // 输入
  inputValue: string;
  
  // 模型
  currentProvider: string;
  currentModel: string;
  
  // Provider 列表（缓存）
  providers: ProviderInfo[];
  models: ModelInfo[];
  
  // UI 状态
  showSettings: boolean;
  isLoadingModels: boolean;
  error: string | null;

  // 连接状态
  bridgeConnected: boolean;                // ← 新增：bridge 是否连接
  lastPingTime: number | null;             // ← 新增：上次心跳时间

  // 动作
  appendMessage: (msg: Message) => void;
  updateAssistantMessage: (id: string, text: string) => void;
  finalizeAssistantMessage: (id: string) => void;
  updateThinkingMessage: (id: string, text: string) => void;
  finalizeThinkingMessage: (id: string) => void;
  updateToolResult: (toolId: string, result?: string, error?: string) => void;
  setInputValue: (v: string) => void;
  sendMessage: () => void;
  clearSession: () => void;
  toggleSettings: () => void;
  switchModel: (ref: string) => Promise<void>;
  loadProviders: () => Promise<void>;
  setBridgeConnected: (connected: boolean) => void;
}
```

### 2.3 数据流

```
C# 后端
  ↓ bridge.postMessage(json)
bridgeService.handleIncoming()
  ↓ 解析消息类型
useChatStore.getState().xxx()    ← 直接调用 store 方法
  ↓ 更新状态
React 重新渲染 MessageList / InputBar / ModelSwitcher
```

```css
text-gray-900 dark:text-gray-100
```

### 4.2 主题策略

- **首版同时实现 light + dark 主题**
- 使用 Tailwind `dark:` 类，通过 `class` 策略切换（在 `<html>` 上添加/移除 `dark` 类）
- 默认跟随系统 `prefers-color-scheme`，可通过设置切换
- 主题选择持久化到 localStorage
- E3D 为深色界面，**默认使用 dark 主题**以保持视觉一致

**实现方式**：
```tsx
// src/hooks/useTheme.ts
function useTheme() {
  const [theme, setTheme] = useState<'light' | 'dark' | 'system'>(
    () => (localStorage.getItem('e3d-theme') as any) || 'system'
  )
  
  useEffect(() => {
    const isDark = theme === 'dark' || 
      (theme === 'system' && window.matchMedia('(prefers-color-scheme: dark)').matches)
    document.documentElement.classList.toggle('dark', isDark)
  }, [theme])
  
  return { theme, setTheme }
}
```
